# AI Finance Assistant Prompt 设计与迭代报告

## 1. 报告说明

本报告记录 AI Finance Assistant 项目中的 Prompt 设计目标、模板管理方式、变量注入方法、结构化输出约束、模型参数选择、风险控制措施和迭代过程。

项目中的 AI 功能主要包括：

1. 消费账单自动分类
2. 中文或英文自然语言账单提取
3. 个性化省钱建议
4. AI 财务顾问流式对话
5. 消费金额异常解释模板
6. 支出预测解释模板

项目使用 Spring AI 的 `ChatClient` 调用 Anthropic 兼容模型接口。Prompt 并未直接写死在 Controller 或业务方法中，而是保存在 PostgreSQL 数据库中，由统一的 Prompt 服务负责读取和渲染。

---

## 2. Prompt 设计目标

本项目的 Prompt 设计主要遵循以下原则。

### 2.1 减少模型幻觉

财务系统涉及用户的真实金额、预算和历史消费，因此 Prompt 必须明确要求模型：

* 不得虚构用户没有提供的消费
* 不得虚构收入、预算或账单
* 数据不足时直接说明
* 不得把预测描述为确定事实
* 不得擅自修改后端已经计算的金额

### 2.2 限制模型输出范围

对于账单分类和结构化提取，模型只能在系统提供的消费分类中选择，不允许自行创建新的分类代码。

当模型返回无法识别的分类代码时，后端会将其替换为 `OTHER`，避免无效分类写入数据库。

### 2.3 将计算与语言生成分离

金额异常判断、预算统计、支出预测和财务健康评分主要由后端确定性逻辑完成。

大语言模型主要负责：

* 理解自然语言
* 提取结构化字段
* 解释已有数据
* 生成自然语言建议
* 进行上下文相关的财务对话

这种设计避免让模型直接承担关键数值计算，提高财务结果的稳定性和可解释性。

### 2.4 提高个性化程度

Prompt 会根据功能注入不同的用户信息，例如：

* 用户所在地区
* 地区物价系数
* 日常消费水平
* 月收入
* 默认月预算
* 当前预算
* 分类消费统计
* 历史账单
* 下月支出预测
* 财务健康评分
* 最近的对话历史

因此，AI 返回的内容不仅依赖用户本次输入，也会结合系统中的真实财务数据。

### 2.5 保证结果可以被程序处理

消费分类和账单提取使用结构化 Java 类型接收模型结果。

Prompt 明确要求模型只返回符合约定结构的 JSON，不返回多余说明。后端再对金额、分类、置信度、日期和币种进行二次校验。

## 3. Prompt 模板管理架构

### 3.1 数据库存储

项目在 PostgreSQL 中建立了 `prompt_template` 表，主要字段包括：

| 字段                     | 作用             |
| ---------------------- | -------------- |
| `code`                 | Prompt 功能编码    |
| `version`              | 模板版本           |
| `system_prompt`        | System Prompt  |
| `user_prompt_template` | User Prompt 模板 |
| `response_schema`      | 预期返回结构         |
| `description`          | 模板用途说明         |
| `active`               | 是否为当前启用版本      |

数据库使用以下唯一约束：

```text
code + version 唯一
```

并使用部分唯一索引保证：

```text
同一种 Prompt 编码最多只能存在一个 active = true 的版本
```

因此，后续升级 Prompt 时可以新增版本，而不必直接覆盖历史模板。

### 3.2 Prompt 编码

当前定义了六个 Prompt 编码：

```text
EXPENSE_EXTRACTION
EXPENSE_CLASSIFICATION
SAVING_ADVICE
FINANCIAL_CHAT
PREDICTION_EXPLANATION
ANOMALY_EXPLANATION
```

### 3.3 模板读取

业务服务不会直接读取 SQL，也不会在 Java 方法中保存完整 Prompt，而是调用：

```java
promptTemplateService.render(
    PromptTemplateService.EXPENSE_EXTRACTION,
    variables
);
```

`PromptTemplateService` 会：

1. 根据 Prompt 编码查询启用的模板。
2. 读取 System Prompt。
3. 读取 User Prompt 模板。
4. 查找 `{{variable_name}}` 格式的变量。
5. 将业务服务提供的数据替换到模板中。
6. 返回渲染后的 Prompt 和模板版本号。

### 3.4 变量渲染

模板变量采用：

```text
{{变量名}}
```

例如：

```text
{{categories}}
{{user_preference}}
{{financial_context}}
{{user_message}}
```

字符串变量直接写入 Prompt。

对象、集合和 DTO 会通过 Jackson `ObjectMapper` 序列化为 JSON，再写入 Prompt。

当模板包含变量，但业务服务没有提供对应值时，系统会直接抛出异常：

```text
Prompt缺少变量：变量名
```

这种处理可以避免变量遗漏后仍向模型发送不完整 Prompt。

## 4. System Prompt 与 User Prompt 的职责划分

### 4.1 System Prompt

System Prompt 主要负责定义长期稳定的行为约束，包括：

* 模型身份
* 任务边界
* 禁止行为
* 输出格式
* 安全要求
* 语言和表达风格

例如，账单分类 System Prompt 规定模型只能使用系统分类，省钱建议 System Prompt 规定不得推荐股票、基金、保险或借贷产品。

### 4.2 User Prompt

User Prompt 负责注入每次请求变化的真实业务数据，例如：

* 本次消费文本
* 分类列表
* 用户偏好
* 财务统计
* 对话历史
* 用户问题

这种拆分使规则与数据相互独立，便于维护和迭代。

## 5. 消费自动分类 Prompt

### 5.1 功能编码

```text
EXPENSE_CLASSIFICATION
```

### 5.2 设计目标

该 Prompt 用于将一条消费记录映射到系统已有分类。

主要约束包括：

1. 只能从系统提供的分类代码中选择。
2. 不允许创建新的分类代码。
3. 综合商品、商家、备注和原始文本判断。
4. 信息不足时使用 `OTHER`。
5. 置信度必须位于0到1之间。
6. 只返回结构化结果。

### 5.3 动态变量

| 变量           | 内容            |
| ------------ | ------------- |
| `categories` | 当前数据库中启用的消费分类 |
| `item_name`  | 消费项目名称        |
| `merchant`   | 商家名称          |
| `note`       | 账单备注          |
| `raw_text`   | 原始自然语言内容      |

### 5.4 期望输出

```json
{
  "categoryCode": "FOOD",
  "confidence": 0.93,
  "reason": "消费项目为咖啡，属于餐饮消费"
}
```

### 5.5 模型参数

```text
模型：小型快速模型
temperature：0.1
maxTokens：500
调用方式：非流式结构化调用
```

分类任务答案范围较小，因此使用快速模型降低延迟和调用成本。

较低的温度用于减少分类结果的随机性。

### 5.6 后端保护

即使模型返回了无效分类，后端也不会直接采用，而是：

1. 将分类代码转为大写。
2. 查询数据库中是否存在该分类。
3. 无效分类自动回退到 `OTHER`。
4. 将置信度限制在0到1之间。
5. 模型未返回置信度时默认使用 `0.5000`。

## 6. 多账单结构化提取 Prompt

### 6.1 功能编码

```text
EXPENSE_EXTRACTION
```

### 6.2 设计目标

该 Prompt 用于把用户输入的中文或英文自然语言转换成一条或多条账单记录。

例如：

```text
今天买了两杯咖啡花了38元，晚上打车花了46元。
```

模型需要拆分为两笔独立消费。

### 6.3 核心约束

1. 不得虚构金额、商品、商家或时间。
2. 一句话中包含多笔消费时必须拆分。
3. 金额必须为正数。
4. 未说明币种时使用 CNY。
5. 未说明时间时使用系统参考时间。
6. 分类只能来自系统提供的分类。
7. 不确定字段使用 `null`。
8. 只输出符合约定结构的 JSON。

### 6.4 动态变量

| 变量                | 内容            |
| ----------------- | ------------- |
| `input_language`  | 中文或英文         |
| `reference_time`  | 系统生成的参考时间     |
| `categories`      | 数据库中的可用消费分类   |
| `user_preference` | 地区、物价、消费水平和币种 |
| `user_text`       | 用户输入或语音识别文本   |

### 6.5 期望输出

```json
{
  "transactions": [
    {
      "itemName": "咖啡",
      "merchant": null,
      "amount": 38,
      "currency": "CNY",
      "quantity": 2,
      "unit": "杯",
      "categoryCode": "FOOD",
      "occurredAt": "2026-06-14T12:00:00Z",
      "note": null,
      "confidence": 0.92
    }
  ]
}
```

### 6.6 模型参数

```text
模型：主模型
temperature：0.1
maxTokens：4096
调用方式：非流式结构化调用
```

账单提取可能一次返回多条记录，因此 Token 上限高于单条分类任务。

低温度用于提高字段提取和 JSON 输出的稳定性。

### 6.7 后端二次校验

模型返回后，后端还会进行以下处理：

* 结果不能为空
* 一次最多接收50条消费记录
* 消费项目不能为空
* 金额必须大于0
* 金额不能超过系统允许的最大值
* 无效分类自动替换为 `OTHER`
* 无效币种自动替换为 `CNY`
* 数量小于等于0时改为 `null`
* 未来时间或无法解析的时间改为参考时间
* 置信度限制在0到1之间
* 置信度低于 `0.6500` 时标记为需要复核
* 分类被后端替换时标记为需要复核

因此，模型结果只是待确认草稿，不会未经检查直接成为正式账单。

## 7. 个性化省钱建议 Prompt

### 7.1 功能编码

```text
SAVING_ADVICE
```

### 7.2 设计目标

该 Prompt 根据系统已经计算出的真实财务数据，生成未来一个月内可执行的省钱建议。

### 7.3 核心约束

1. 不得虚构收入、账单或消费行为。
2. 尊重用户所在地区、预算和消费习惯。
3. 不使用羞辱、说教或制造焦虑的语言。
4. 优先分析最有改善空间的消费分类。
5. 建议应当能够在未来一个月内执行。
6. 不推荐股票、基金、保险或借贷产品。
7. 金额统一使用人民币。
8. 使用中文 Markdown。

### 7.4 动态变量

| 变量                    | 内容                 |
| --------------------- | ------------------ |
| `user_preference`     | 地区、物价、消费水平、收入和默认预算 |
| `budget_summary`      | 本月预算与预算使用情况        |
| `expense_summary`     | 本月支出摘要             |
| `category_statistics` | 各消费分类的金额和占比        |
| `prediction`          | 下月支出预测             |
| `health_score`        | 财务健康评分             |

### 7.5 模型参数

```text
模型：主模型
temperature：0.2
maxTokens：2048
调用方式：非流式文本生成
输出格式：中文 Markdown
```

建议任务需要一定语言表达能力，因此温度略高于结构化提取任务，但仍保持在较低水平，以减少不稳定建议。

### 7.6 保存方式

生成结果不能为空。

有效结果会保存到 `saving_advice` 表，并关联：

* 当前用户
* 目标月份
* 财务健康评分
* 建议优先级
* Markdown 内容
* 建议状态

用户可以将建议标记为：

```text
ACTIVE
ADOPTED
DISMISSED
```

## 8. AI 财务顾问 Prompt

### 8.1 功能编码

```text
FINANCIAL_CHAT
```

### 8.2 设计目标

该 Prompt 支持类 ChatGPT 的多轮个人财务咨询。

模型只能根据系统提供的用户数据和对话历史回答问题。

### 8.3 核心约束

1. 不得虚构账单、预算、收入或历史行为。
2. 金额统一使用人民币。
3. 回答应清楚、具体，符合中文表达习惯。
4. 可以分析预算、消费结构、异常支出、预测和健康评分。
5. 不提供保证收益的投资建议。
6. 不替代会计、税务、法律或持牌金融服务。
7. 数据不足时直接说明，不得猜测。
8. 使用 Markdown 输出。

### 8.4 动态变量

| 变量                     | 内容               |
| ---------------------- | ---------------- |
| `user_preference`      | 地区、消费水平、收入、预算和币种 |
| `financial_context`    | 当前月份财务统计数据       |
| `conversation_history` | 最近的历史消息          |
| `user_message`         | 用户当前问题           |

### 8.5 对话历史控制

系统最多读取最近20条消息。

消息会按照时间重新排列后注入 Prompt，格式包括：

```json
[
  {
    "role": "USER",
    "content": "我这个月餐饮支出高吗？"
  },
  {
    "role": "ASSISTANT",
    "content": "根据当前数据……"
  }
]
```

限制历史数量可以防止 Prompt 无限增长，减少 Token 消耗。

### 8.6 模型参数

```text
模型：主模型
temperature：0.2
maxTokens：4096
调用方式：SSE 流式生成
输出格式：Markdown
```

### 8.7 SSE 事件

AI 顾问通过以下事件逐步向前端返回内容：

| 事件      | 作用            |
| ------- | ------------- |
| `start` | 表示生成开始        |
| `delta` | 返回一个文本片段      |
| `done`  | 表示生成完成并返回消息编号 |
| `error` | 表示生成失败        |

后端在流式输出过程中使用 `StringBuilder` 保存完整结果。

只有生成完成后，完整回答才会写入数据库。

## 9. 异常金额解释 Prompt

### 9.1 功能编码

```text
ANOMALY_EXPLANATION
```

### 9.2 设计目标

后端先根据价格规则、地区物价、用户偏好和历史价格完成异常判断。

该 Prompt 只负责将判断结果转换成自然、礼貌的提醒，不参与重新计算。

### 9.3 变量

```text
item_name
amount
reference_range
spending_level
price_index
personal_history
anomaly_reason
```

### 9.4 当前状态

该模板已经保存在数据库中，但当前版本的异常提醒主要由后端确定性代码直接生成，尚未在业务服务中调用该 Prompt。

保留该模板是为了后续升级为更自然的异常解释，但不会影响当前异常检测功能。

## 10. 支出预测解释 Prompt

### 10.1 功能编码

```text
PREDICTION_EXPLANATION
```

### 10.2 设计目标

支出预测金额由后端统计算法计算。

该 Prompt 只能解释预测结果，不得重新计算或修改预测值。

### 10.3 变量

```text
target_month
predicted_amount
lower_bound
upper_bound
algorithm
based_on_months
monthly_history
```

### 10.4 当前状态

该模板已经保存在数据库中，但当前预测解释主要由 `PredictionService` 根据数据量和算法类型生成固定说明，尚未调用大语言模型。

这种实现可以避免为简单解释增加模型调用成本，同时确保预测说明与实际算法完全一致。

## 11. 模型选择策略

项目使用两类模型配置。

### 11.1 小型快速模型

适用于：

```text
EXPENSE_CLASSIFICATION
```

原因：

* 输入较短
* 输出字段较少
* 任务范围明确
* 更关注响应速度和调用成本

### 11.2 主模型

适用于：

```text
EXPENSE_EXTRACTION
SAVING_ADVICE
FINANCIAL_CHAT
```

原因：

* 需要处理较长文本
* 可能提取多条账单
* 需要理解复杂财务上下文
* 需要生成自然语言或 Markdown
* 需要处理多轮对话历史

模型名称通过环境变量配置，不写死在代码中：

```text
ANTHROPIC_MODEL
ANTHROPIC_SMALL_FAST_MODEL
```

## 12. Prompt 迭代过程

以下内容记录的是本项目的设计迭代过程，而不是单纯的模板版本编号。

当前数据库中的正式模板版本均为 `version = 1`。

### 12.1 第一阶段：直接向模型发送用户文本

最初的核心目标是让模型理解用户输入，例如：

```text
午饭35，打车20
```

这一方式存在以下问题：

* 模型可能返回自然语言而不是 JSON
* 多笔消费不一定被正确拆分
* 可能生成系统中不存在的分类
* 金额、时间和币种格式不稳定
* 无法追踪 Prompt 版本

### 12.2 第二阶段：增加角色和明确规则

Prompt 被拆分为 System Prompt 和 User Prompt。

System Prompt 负责规定：

* 模型角色
* 禁止虚构
* 输出规则
* 分类范围
* 安全边界

User Prompt 只负责放入本次请求的数据。

这一变化使长期规则与动态业务数据分离。

### 12.3 第三阶段：加入结构化输出约束

分类和账单提取增加了明确字段，并在数据库中保存 `response_schema`。

业务服务使用：

```java
.entity(ModelResult.class)
```

将模型结果转换为 Java Record。

这种方法减少了前端解析自然语言结果的复杂度。

需要说明的是，当前 `response_schema` 主要用于模板管理、设计说明和版本记录；`PromptTemplateService` 目前不会自动把该字段传给模型。实际结构化转换依赖 Prompt 中的 JSON 约束和 Spring AI 的 `.entity(...)`。

### 12.4 第四阶段：动态注入分类和用户偏好

不再在 Prompt 中固定写死分类，而是从数据库读取当前启用的分类并动态注入。

账单提取还加入：

* 地区
* 物价系数
* 消费水平
* 默认币种

省钱建议和 AI 顾问进一步加入：

* 月收入
* 月预算
* 财务统计
* 支出预测
* 健康评分

### 12.5 第五阶段：增加后端二次校验

Prompt 约束不能完全替代程序校验，因此项目增加了：

* 无效分类回退
* 置信度范围修正
* 金额范围校验
* 时间格式校验
* 币种格式校验
* 低置信度人工复核
* 空结果判断
* 最大记录数量限制

这一阶段形成了：

```text
Prompt 约束 + Java 类型映射 + 后端业务校验
```

三层保护机制。

### 12.6 第六阶段：加入财务上下文和流式对话

AI 顾问不再只根据用户当前一句话回答，而是注入：

* 用户偏好
* 当前财务统计
* 最近20条历史消息
* 当前问题

同时采用 SSE 流式返回，提升等待过程中的交互体验。

## 13. 幻觉与安全风险控制

### 13.1 财务数据幻觉

风险：

模型可能编造不存在的账单、收入或预算。

处理方式：

* System Prompt 明确禁止虚构
* 只向模型提供后端查询出的真实数据
* 数据不足时要求直接说明
* 数值计算主要由后端完成

### 13.2 无效分类

风险：

模型可能生成数据库中不存在的分类代码。

处理方式：

* 动态传入可用分类
* Prompt 禁止创建新分类
* 后端查询分类是否存在
* 无效结果回退到 `OTHER`

### 13.3 JSON 格式错误

风险：

模型可能返回 Markdown 代码块或额外说明。

处理方式：

* Prompt 要求只输出 JSON
* Spring AI 将结果映射到 Java Record
* 映射或解析失败时统一抛出 AI 处理异常

### 13.4 不合理金额

风险：

模型可能识别出负数、0或过大金额。

处理方式：

* Prompt 要求金额为正数
* 后端再次校验金额范围
* 不合理结果不会进入正式账单

### 13.5 投资和专业服务风险

风险：

财务顾问可能给出保证收益或专业法律、税务结论。

处理方式：

* 禁止保证收益的投资建议
* 省钱建议不推荐股票、基金、保险和借贷产品
* 明确说明不能替代持牌金融、会计、税务或法律服务

### 13.6 Prompt 注入

风险：

用户输入可能包含“忽略之前规则”等指令。

当前防护包括：

* 核心规则放在 System Prompt
* 用户文本作为独立变量放入 User Prompt
* 后端不会执行模型生成的 SQL 或代码
* 模型输出必须经过类型转换和业务校验
* AI 无权直接修改用户数据

后续还可以增加输入内容分隔符、注入检测和输出内容审查。

## 14. 当前不足

### 14.1 `response_schema` 尚未动态参与调用

虽然数据库保存了 JSON Schema，但当前 `PromptTemplateService` 只读取：

* System Prompt
* User Prompt
* 模板版本

尚未将 `response_schema` 动态传给 Spring AI。

当前结构化输出主要依赖 `.entity(...)` 和 Prompt 中的格式要求。

### 14.2 两个解释模板尚未接入

以下模板已经建立，但当前业务流程没有调用：

```text
ANOMALY_EXPLANATION
PREDICTION_EXPLANATION
```

目前对应说明由后端确定性代码生成。

### 14.3 尚无 Prompt 后台管理页面

Prompt 目前通过 Flyway 初始化，尚未提供管理员页面用于：

* 新增模板版本
* 切换启用版本
* 对比历史版本
* 灰度测试
* 回滚 Prompt

### 14.4 尚未保存完整 Token 使用量

数据库中的 AI 消息结构已经预留：

```text
prompt_tokens
completion_tokens
metadata
```

但当前业务服务尚未完整记录每次调用的 Token 消耗。

### 14.5 尚未建立自动化 Prompt 评测集

当前没有单独的 Prompt Evaluation 数据集。

后续可以建立典型输入样本，对以下指标进行自动比较：

* 分类准确率
* 金额提取准确率
* 多账单拆分准确率
* JSON 格式成功率
* 低置信度识别率
* 建议可执行性
* 幻觉率

## 16. 总结

本项目没有将 Prompt 简单地写在 Controller 或 Service 方法中，而是建立了数据库化、可版本管理的 Prompt 模板体系。

系统通过以下方式提高 AI 功能的可靠性：

```text
数据库 Prompt 模板管理
+ System Prompt 行为约束
+ 动态业务变量注入
+ 结构化 Java 类型映射
+ 后端二次校验
+ 用户人工确认
```

其中：

* 自动分类使用快速模型完成低成本判断
* 自然语言账单提取使用主模型完成结构化转换
* 省钱建议结合预算、预测和健康评分生成
* AI 顾问结合财务摘要和对话历史进行 SSE 流式回答
* 关键金额计算仍由后端确定性算法完成

该设计在生成能力、财务安全、结果可控性和系统维护性之间取得了较好的平衡。
